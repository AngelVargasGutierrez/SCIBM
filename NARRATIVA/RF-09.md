# RF-09 – Modificar solucionario y calibración de examen maestro.

## a) RF-09 – Modificar solucionario y calibración de examen maestro.

| Campo | Detalle |
|---|---|
| **Caso de uso** | UC-09 – Modificar solucionario y calibración de examen maestro. |
| **Actores** | Docente |
| **Propósito** | Permitir al docente regresar a la configuración de un examen ya registrado para corregir el enunciado, puntaje, respuestas de las preguntas o ajustar la posición de los pines de calibración visual. |
| **Tipo** | Obligatorio (X) / Opcional ( ) |
| **Requisito ID (RF)** | RF-009 |
| **Versión** | 1.0 |
| **Descripción** | El docente accede a la vista de "Editar Solucionario" eligiendo la pestaña de Datos o Calibración. En Datos, puede modificar en línea textos, respuestas y puntajes. En Calibración, puede mover los pines visuales sobre el PDF. Ambas acciones guardan los cambios de forma asíncrona (AJAX) sin recargar la página completa, validando que el docente sea el propietario del curso. |
| **Precondición** | El usuario debe ser el propietario del curso y el examen debe existir en la base de datos. |
| **Postcondición** | Las modificaciones en el solucionario o en las coordenadas de corrección se actualizan y persisten en la base de datos. |

---

### Curso normal de eventos

| # | Usuario | Sistema |
|---|---|---|
| 1 | El docente, queriendo alterar un examen antiguo, navega hasta el panel de control del examen y acciona la opción "Modificar Solucionario". | |
| 2 | | El sistema web intercepta la petición HTTP de tipo GET a la ruta `EditSolucionario` y ejecuta una búsqueda en la base de datos para localizar toda la estructura jerárquica que engloba ese examen. |
| 3 | | El sistema contrasta rigurosamente el correo del docente en sesión contra el `DocenteEmail` oficial adscrito a ese ciclo y materia. Una vez aprobada la validación, inicia la carga de la pantalla. |
| 4 | | El sistema estructura y renderiza un entorno visual configurado mediante contenedores de navegación (Pestañas/Tabs) posados encima de un fondo neutral blanco `#ffffff`. |
| 5 | | El sistema dibuja la botonera de Pestañas con un cinturón o base gris perla `#f8f9fa`. De forma intencionada, la pestaña que el usuario está viendo actualmente ("Datos") carece de gris y usa fondo completamente blanco `#ffffff` junto a un grueso ribete azul `#0d6efd` de `3px` cruzando su parte superior. |
| 6 | El docente desplaza el cursor y hace clic sobre el cuadro de texto (input) del puntaje de la pregunta que desea reajustar. | |
| 7 | | El sistema dispara los desencadenantes de diseño de estado "focus", los cuales transforman drásticamente el delineado del input, pasando de un gris estándar y plano `#ced4da` a un trazo en tono azul encendido `#86b7fe`, complementado por una suave sombra brillante color celeste rodeando los bordes `box-shadow: 0 0 0 0.25rem rgba(13,110,253,.25)`. |
| 8 | El docente escribe la nueva cifra de puntaje usando el teclado y, al estar conforme, desliza el ratón para dar clic en el botón "Guardar Cambios". | |
| 9 | | El sistema captura la interacción manual en este componente flotante. El botón, siempre anclado fijamente en la esquina inferior derecha de la ventana visual, destaca gracias a su paleta azul primary `#0d6efd`, los bordes redondeados ergonómicos y un gráfico vectorizado de disquete blanco puro. |
| 10 | | El sistema moviliza de inmediato la lógica Javascript residente en el lado del cliente (Frontend), la cual extrae minuciosamente todos los valores editados de los formularios del DOM, condensándolos y empaquetándolos en un arreglo nativo JSON. |
| 11 | | El sistema ejecuta una conexión encubierta u oculta en segundo plano (asíncrona mediante tecnología AJAX/Fetch), dirigiendo el paquete estructurado al punto de acceso API `SaveSolucionarioData`. |
| 12 | | El bloque lógico del sistema (Backend) intercepta los parámetros, abre un hilo hacia la base de datos Entity Framework e inicializa la recolección de las preguntas vinculadas a este examen en particular. |
| 13 | | El sistema realiza un barrido cruzando los datos entrantes frente a los preexistentes, sobreescribiendo en memoria aquellos que varíen, como la columna de `Puntaje` y el `Enunciado`. |
| 14 | | El sistema finaliza el salvamento asíncrono confirmando la escritura masiva y propagando un código de estado correcto de vuelta hacia el explorador web. |
| 15 | | La interfaz gráfica recepta el acierto e invoca a las rutinas de Notificación (Toast), provocando que un panel rectangular resbale fluidamente (animación css slide-in) desde el borde extremo derecho adentrándose a la pantalla principal. |
| 16 | | El sistema expone visualmente esta pequeña notificación construyéndola a partir de un fondo de color verde suave `#d1e7dd`, envuelta en un marco delgado verde oscuro `#0f5132`. |
| 17 | | Finalmente, dentro del recuadro, el sistema inyecta el texto afirmativo "Respuestas y puntajes actualizados correctamente" rotulado en una fuente nítida de la misma tonalidad oscura `#0f5132`. |

### Flujo Alterno (Excepción: Sesión Caducada o Sin Permisos)

| # | Usuario | Sistema |
|---|---|---|
| 1 | El docente intenta modificar un solucionario pero su sesión ha expirado tras horas de inactividad, o intenta acceder a la fuerza a un curso que no le pertenece. | |
| 2 | | El sistema recibe la petición HTTP asíncrona y verifica el Token de Autenticación del usuario. |
| 3 | | El sistema determina que el usuario actual no cuenta con privilegios sobre este curso (`DocenteEmail` no coincide con el dueño). |
| 4 | | El sistema bloquea irrevocablemente cualquier guardado o modificación en la base de datos. |
| 5 | | El sistema redirige automáticamente al usuario a la pantalla principal de Inicio de Sesión o le muestra un aviso gigante de "Acceso Denegado". |
